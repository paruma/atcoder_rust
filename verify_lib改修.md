> .gemini/skills/code-verifier や verify_lib.py は使いにくいですか？改善点はありますか？
✦ .gemini/skills/code-verifier と verify_lib.py
  を実際に使用した経験から、「非常に強力で頼りになるが、あと一歩の改善でさらに使いやすくなる
  」と感じました。

  特に今回の coordinate_compression.rs の件で直面した課題をもとに、改善点を提案します。


  1. 良かった点（使いやすい点）
   * 「これさえ流せば安心」という一貫性:
     テスト、カバレッジ、静的解析、フォーマットを一括でチェックできるため、ライブラリの品質
     維持に対する心理的ハードルが下がります。
   * 行番号の指摘: 最初に 61 行目を指摘してくれたおかげで、Bound::Excluded
     のテスト漏れに即座に気づけました。


  2. 改善が必要と感じた点（使いにくい点）
   * カバレッジの「ノイズ」:
      llvm-cov
  の標準出力がそのまま出るため、指定したファイル以外の情報（クレート内の全ファイル）が混ざり
  、本当に見たい情報（ターゲットファイルの詳細）が埋もれてしまいます。
   * 「行内分岐（Region）」への対応:
      今回のように「Missed Lines: 2」と言いつつ行番号が出ないケースがありました。これは
  llvm-cov
  が「行全体」ではなく「行の一部の分岐」が未実行だと判断しているためですが、ツール側で「どの
  メソッド内のどのあたりか」というヒントをもっと出せると助かります。
   * パス指定の二度手間:

  ファイルパス（src/mylib/...）からモジュールパス（algorithm::...）への変換を人間（またはエ
  ージェント）が行う必要があります。ファイルパスを直接渡せるとよりスムーズです。

  3. 具体的な改善案（verify_lib.py のアップデート）


   1. ターゲットファイルのフィルタリング強化:
      llvm-cov 実行時に --filter-files <ファイル名>
  を付与するか、出力のパース時に対象ファイル以外の行を完全に隠蔽する。
   2. 未実行コードのテキスト表示:
      行番号だけでなく、その行のソースコード自体を 1〜2 行表示してくれると、わざわざ
  read_file しなくて済むため、エージェントにとっても人間にとっても非常に助かります。
   3. HTMLレポートへの導線:
      カバレッジが 100% でない場合、詳細を確認するための HTML
  レポートの場所を表示する、あるいは HTML を生成して「ここを見て」と促す機能。
   4. 自動修正（Autofix）オプション:
      cargo fmt や一部の clippy 警告について、スクリプト側で cargo fmt や cargo clippy --fix
  を実行して、そのまま正常な状態に持っていくオプションがあると、コンテスト中でも使いやすいで
  す。


  結論
  現状でも、「嘘の改善（計算量の誤解など）」を暴き、テスト漏れを指摘してくれる非常に優秀なガ
  ードレールとして機能しています。
  上記のような「出力の絞り込み」と「情報の具体化」が行われれば、エージェントとの相性はさらに
  良くなる（思考ステップをさらに短縮できる）はずです。
