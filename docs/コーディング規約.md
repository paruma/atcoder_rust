# Rust ライブラリ開発 コーディング規約

本ドキュメントは、AtCoder 用 Rust ライブラリ（`mylib`）の開発における品質維持と、AI エージェントによる効率的な実装・メンテナンスのための指針を定義する。

## 1. 設計原則

### 1.1. 抽象代数的設計 (Abstract Algebraic Design)
特定の具象型やドメインに依存せず、計算に必要な最小限の「代数的性質」のみを要求する。
- **ドメイン依存の排除**: `From<i64>` 等の使用を避け、型 `T` が整数であるといった強い仮定を排除する。
- **単位元の生成**: `0.into()` の代わりに `std::iter::empty::<T>().sum()` を用いるなど、代数系内部の演算ルールから導き出す。

### 1.2. 構造的対称性の維持 (Structural Symmetry)
対になる概念（最小/最大など）を扱うファイル・関数群は、完全にミラー構造を維持する。
- 行数、空行、インポート順、メソッドの定義順を一致させる。

### 1.3. スニペット設計
`cargo-snippet` による展開を考慮し、依存関係を最小化する。
- **外部依存の配置**: `std` 等の外部依存は `#[snippet]` が付与された `mod` ブロック内部で `use` する。
- **ローカル依存**: `use super::Symbol;` を用い、`include` 属性で紐付ける。`use crate::` や `use super::*` は禁止。

## 2. 実装規約

### 2.1. エラーハンドリング
- **異常系のパニック**: メソッドの前提条件（インデックス範囲など）を外れる入力に対しては、妥協してデフォルト値を返すのではなく、速やかに `panic!` させる。

### 2.2. 計算量表記
- **LaTeX 形式**: ドキュメントコメント内では `$O(N \log N)$` のように LaTeX 形式で記述する。
- **変数の統一**: 実装上の変数名（`size`, `len`, `entries.len()` 等）にかかわらず、データサイズを表す記号には原則として $N$ を使用する。

### 2.3. 可視性
- ライブラリ内部でのみ使用される補助構造体（例：`InternalFenwickTree`）は、外部に露出させないよう**非公開（可視性修飾子なし）**として定義する。スニペットとして `main.rs` 等の単一クレート内に展開された際、他所から意図せずアクセス可能になるのを防ぐため、`pub(crate)` も使用しないこと。

## 3. ドキュメント規約

### 3.1. ドキュメントコメント (Doc comments)
- **形式**: `///` を使用する。
- **構造**: 
    1. 1行目の要約。
    2. 空行（必須）。
    3. 詳細な説明、計算量、パニック条件。
- **言語**: 日本語で記述する。

## 4. テスト規約

### 4.1. ユニットテスト
- **正常系**: 基本的な追加、削除、検索を網羅。
- **エッジケース**: 空の状態、要素数 1、サイズ 0 の初期化、重複要素、境界値。
- **パニックテスト**: `#[should_panic]` を用い、不正な入力で正しく落ちることを確認。

### 4.2. ランダムテスト (Random Testing)
- **比較対象**: 素朴な実装（Naive Implementation）や、既に検証済みの信頼できる既存ライブラリを「正解（Oracle）」とし、ランダムな操作群に対して実行結果が完全に一致することを確認する。
- **入力値の密度設計（重要）**: 入力値の生成範囲（ドメイン）をあえて狭く設定することで、**値の衝突（Collision）**や**境界条件の発生頻度**を高める。これにより、要素の重複、特定条件での削除、空の状態への遷移といった、広い入力空間では滅多に発生しないエッジケースを効率的に網羅する。
- **異常系・境界外の網羅**: 正常な値だけでなく、意図的に境界外の引数や無効なデータ状態での操作をテストに含め、失敗系（`None` の返却や `panic!`）が正しく機能することを検証する。
- **属性**: 必ず `#[ignore]` を付与し、通常のビルド・テストサイクルを妨げないようにする。
- **rand クレート**: バージョン `0.9.2` を使用。`SmallRng::from_os_rng()` で初期化し、`rng.random_range()` を使用する。

## 5. 検証プロトコル

### 5.1. 自動検証の必須化
`src/mylib` 配下を編集した後は、必ず統合検証スクリプトを実行し、品質を保証する。
- **コマンド**: `python3 .gemini/skills/code-verifier/scripts/verify_lib.py <path>`
- **合格基準**:
    - 全テストの通過。
    - カバレッジが十分であること（主要ロジックは 100% を目指す。`panic!` 内部などは不問）。
    - Clippy 警告（`-D warnings`）がないこと。
    - `cargo fmt` による整形済みであること。
