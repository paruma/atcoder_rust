# Contest Retrospective

## 概要

このスキルは、AtCoder 等のコンテスト終了後に、自分の提出コードを客観的にレビューし、学びを最大化するための高品質なレポート（Markdown）を作成します。

## ワークフロー

振り返りを行う際は、以下の手順に従ってください。

### 1. ディレクトリとコードの特定
指定されたコンテストディレクトリ（例：`src/contest/YYYY/abcXXX`）内の全ソースコードを読み込みます。
ただし、ダミー実装のままの問題については振り返りの対象外とします。

### 2. 問題情報の取得
`atcoder-fetcher` スキルを有効化し、対象コンテストの問題文を取得します (問題が複数ある場合は並列実行して取得するとよい)
- 問題ごとの制約、コーナーケース、要件を把握します。

### 3. コード分析
ユーザーの実装を以下の観点で分析を行います。
- **ロジック**: 解法の概要、アプローチの整理。
- **計算量**: 最悪ケースのオーダー（例：$O(N \log N)$）が適切か。
- **実装の質**: コーナーケースへの配慮、Rust らしいイディオムの活用。
- **洞察・気づき**: 実装やアルゴリズムを深く読んで得られる気づきを積極的に言語化する。数学的構造（既知の定理・表現との対応）、アルゴリズムのパターン、問題の本質的な言い換え、データ構造の選択理由など、表面的なコード説明にとどまらない観点を含める。

### 4. リファクタリング案の作成
アルゴリズム自体は変えずに、より Rust らしく、簡潔で実装のしやすいコードを提案します。
- ライブラリ（標準を含む）のメソッドやイテレータを用いた具体的な実装例（ソースコード）を提示します。

リファクタリング案は、実際にコードで示してください。
妥協はせず、分量は多めにしてください
リファクタリング案は一つにする必要はありません。複数ある場合は複数提示してください

### 5. レポートの生成と保存
`[コンテスト名].md` という名前で、以下のフォーマットに従ってレポートを生成し、ルートディレクトリにファイルとして書き出します。

## レポートフォーマット

```markdown
## [問題名 (例: A - Welcome to AtCoder)]

### 問題概要
何を求める問題かを 1〜2 行で簡潔に記述します。

### 実装の評価
- **ロジック**: 解法の概要、独自の工夫など。
- **計算量**: オーダーと実行時間の余裕について。
- **考慮点**: エッジケースへの対応状況。

### 洞察・気づき
実装を深く読んで得られる気づきを記述します。
数学的構造・定理との対応、アルゴリズムのパターン、問題の本質的な言い換えなど。

### リファクタリング案・別解

```

## 注意点

- BTree について言及する場合は `.gemini/knowledge/btree.md` を読んでください
- iter の代わりに into_iter を勧める必要はありません。
- 一度作成したレポートをユーザの指示を下に再編集する場合は、元々記載の内容を簡略化してはいけない。
- i64::MAX などでの初期化は誤って参照した際にすぐ気付ける値である。そのため、i64::MAX を 0 などにする提案は不要
