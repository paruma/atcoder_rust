# Contest Retrospective

## 概要

このスキルは、AtCoder 等のコンテスト終了後に、自分の提出コードを客観的にレビューし、公式解説との比較を通じて学びを最大化するための高品質なレポート（Markdown）を作成します。

## ワークフロー

振り返りを行う際は、以下の手順に従ってください。

### 1. ディレクトリとコードの特定
指定されたコンテストディレクトリ（例：`src/contest/YYYY/abcXXX`）内の全ソースコードを読み込みます。

### 2. 問題情報と解説の取得
`atcoder-fetcher` スキルを有効化し、対象コンテストの問題文と公式解説（Editorial）を取得します。
- 問題ごとの解法、計算量、コーナーケース、期待されるアルゴリズムを把握します。

### 3. 比較分析
ユーザーの実装と公式解説を比較し、以下の観点で分析を行います。
- **ロジック**: 解法が公式と同じか、あるいは異なるアプローチ（別解）か。
- **計算量**: 最悪ケースのオーダー（例：$O(N \log N)$）が適切か。
- **実装の質**: コーナーケースへの配慮、Rust らしいイディオムの活用。

### 4. リファクタリング案の作成
アルゴリズム自体は変えずに、より Rust らしく、簡潔で可読性の高いコード（Clean Rust）を提案します。
- 標準ライブラリのメソッドの活用。
- イテレータを用いた宣言的な記述への変換。

### 5. レポートの生成
`[コンテスト名].md` という名前で、以下のフォーマットに従ってレポートを出力します。

## レポートフォーマット

```markdown
## [問題名 (例: A - Welcome to AtCoder)]

### 問題概要
何を求める問題かを 1〜2 行で簡潔に記述します。

### 実装の評価
- **ロジック**: 公式解説との一致度、独自の工夫など。
- **計算量**: オーダーと実行時間の余裕について。
- **考慮点**: エッジケースへの対応状況。

### リファクタリング案・別解
- より Clean な Rust 実装例や、パフォーマンスが向上する記述方法を提案します。
- 必要に応じて、公式解説にある別解の実装方針も示します。

## 総評
- コンテスト全体の実装の質や、今回学んだ新しいテクニックのまとめ。
- 次回以降に活かせる汎用的な知見（ライブラリ化すべき処理など）。

```
